# typingtrainer

A local-first browser typing trainer focused on QWERTY touch typing, muscle memory, and full keyboard control.

This project is being built first for personal use, but the goal is to make it clean and mature enough to share later.

What is working now:

- guided browser-based lessons with real keystroke scoring
- adaptive practice generated from stored weakness data
- local profiles and local session history
- stats for weak keys, finger zones, substitutions, timing hesitation, and heatmaps
- staged beginner lesson ladder plus coding crossover packs

## Vision

The trainer should help the user stop looking at the keyboard and build reliable, repeatable muscle memory across the full keyboard.

The primary goal is not raw speed. Speed is expected to follow from:

- accuracy
- correct fingering
- consistent muscle memory
- confident use of letters, symbols, modifiers, and code syntax

## Product Direction

`typingtrainer` is a desktop browser app with local-first storage.

Why browser-first:

- better keyboard event visibility than a terminal UI
- can detect physical key identity such as `KeyboardEvent.code`
- can distinguish modifier key side such as left/right `Shift` via `event.location`
- stays mostly OS-independent
- works well as a local app without requiring a backend

Primary target for v1:

- Firefox first
- ISO physical keyboards
- QWERTY layout only

## Stack

The current scaffold is intentionally simple and local-first.

Selected stack:

- Vite
- React
- TypeScript
- Zustand for session and UI state
- Dexie on top of IndexedDB for local profiles and stats
- Vitest and Testing Library for unit and component tests
- Playwright for end-to-end tests, with Firefox coverage prioritized

Deliberate non-choice:

- no Next.js or SSR-first framework for v1

The typing engine does not live inside React component state. Hot-path logic stays in plain TypeScript modules so keyboard capture, lesson matching, error classification, and statistics remain testable and independent from UI rendering.

## Phase 2 Engine Design

The Phase 2 typing engine is built as a plain TypeScript layer, with React only rendering the resulting state.

Current engine rules:

- `keydown` events are scored
- `keyup` events are still captured so modifier state, especially left/right `Shift`, stays accurate
- lesson runners ignore paste-style input and score physical key events instead
- strict mode does not advance on wrong keys
- guided and relaxed modes advance while logging mistakes
- backspace is disabled in strict mode and enabled in guided/relaxed modes
- shift-side technique is enforced for strict technique lessons
- acronym and all-caps tokens such as `README` are exempt from shift-side enforcement
- adjacent same-hand misses can be tagged as likely finger drift
- long pauses inside prompts can be tagged as timing hesitation
- prompts auto-advance when complete
- completed lessons write local session summaries to IndexedDB

Current Phase 2 vertical slice:

- interactive guided lesson runner
- live accuracy and WPM
- wrong-key and wrong-shift-side classification
- likely-wrong-finger heuristics for neighboring-key drift
- local session summary persistence
- stats pages backed by stored session data
- target-key and target-finger analytics derived from stored session summaries
- common substitution tracking derived from expected-to-actual key pairs
- timing hesitation tracking derived from long-pause patterns inside prompts
- recent session progress cards and a keyboard heatmap view
- adaptive recommendations backed by stored session summaries
- generated adaptive practice sessions that turn recent weakness patterns into runnable drills
- production-only service worker registration for offline reloads after first load

## Who It Is For

v1 is designed first for one user who is still building touch-typing discipline on QWERTY:

- partial confidence already exists
- touch typing is not yet consistent
- the user often falls back to looking at the keyboard
- the user wants deep keyboard control for everyday writing, coding, and future VR-heavy workflows

The trainer may later become shareable and more general-purpose.

## Core Principles

- Accuracy before speed
- Fingering before comfort-habit shortcuts in foundational lessons
- Muscle memory before leaderboard-style optimization
- Corrective coaching over passive measurement
- Local-first by default
- Offline-capable after first load
- Coding practice must support typing goals, not replace them

## Practice Modes

### 1. Guided Lessons

A structured beginner path that teaches:

- home row and finger mapping
- letter groups
- numbers and punctuation
- brackets and symbols
- capitalization technique
- modifier usage
- prose typing
- code typing

The guided path is now staged and ordered:

- Stage 1: home row and finger anchors
- Stage 2: top row, bottom row, and number-row reach
- Stage 3: capitals, punctuation, brackets, and modifier control
- Stage 4: English-first prose carryover, then German support
- coding crossover lessons unlock after the core technique path

### 2. Adaptive Training

The trainer analyzes recurring mistakes and generates targeted drills based on:

- weak fingers
- weak keys
- common substitutions
- repeated timing issues
- punctuation/symbol difficulty
- modifier misuse
- code-specific syntax errors

The first implementation is deterministic and local-first:

- it builds a short adaptive session from recent stored summaries
- it prioritizes shift-side mistakes, timing hesitation, weak target keys, symbol trouble, and weak finger zones
- it stays offline-friendly and does not require AI generation

### 3. Free Practice

The user can practice with:

- curated built-in text
- curated built-in code snippets
- generated drills from personal mistake history

## Shift-Key Policy

The app should treat `Shift` usage as part of technique training.

Baseline rule:

- in strict technique lessons, enforce correct shift-side usage for normal capitals and symbols
- in acronym or all-caps contexts, allow same-side hold behavior such as `README`
- in free practice, warn by default instead of failing unless strict mode is explicitly enabled

## Coding Teacher

The coding teacher is a typing-focused training mode, not a full programming education product.

Its role is to improve comfort and accuracy with code-shaped typing.

### V1 Coding Teacher Includes

- syntax drills for Python, MicroPython, and C
- bracket and delimiter drills
- indentation and formatting drills
- identifier typing drills
- short realistic code snippets
- small full functions/programs
- adaptive code drills based on typing mistakes
- light concept prompts that support meaningful typing

### V1 Coding Teacher Does Not Include

- code execution
- automated judging of programming solutions
- project-based coding education
- full AI pair programming
- deep language tutoring workflows

## Gamification

The motivational layer should stay useful and not distract from technique.

V1 includes:

- streaks
- level progression
- achievements

These systems should reinforce disciplined repetition, not reward sloppy speed.

## Storage Model

v1 uses local-only named profiles.

Profiles should store things like:

- lesson progress
- statistics history
- finger/key weakness patterns
- preferred strictness
- coding-language preferences
- unlocked achievements

No backend is required for core use.
